# 评审与审批规则

> 本文档用于约束“测试整理”“结构性重构”“文档补充”这类低风险变更的评审与审批方式。

## 1. 目的

这类变更的核心目标不是引入新功能，而是降低评审成本、改善可维护性、保持代码结构清晰。

审批时应优先回答以下问题：

1. 变更是否真的在降低理解成本。
2. 变更是否保持了运行时行为与对外契约稳定。
3. diff 是否足够小、边界是否足够清楚，便于快速审阅。

## 2. 适用范围

适用于以下类型的改动：

- 测试模块重组、测试命名整理、测试分层调整
- 长函数拆分、私有 helper 提取、职责边界收敛
- 审批文档、说明文档、约束文档补充
- 不改变外部行为前提下的可读性优化

不适用于以下类型的改动：

- 新增业务能力
- 修改业务规则、状态机语义或调度策略
- 修改 API、Tauri command、事件载荷或共享数据结构契约
- 修改用户可见错误文案、日志 key 或可观测性语义
- 为了重构而扩大生产代码可见性

## 3. 核心规则

### 3.1 行为保持优先

结构性调整默认应视为“行为保持”变更。

要求：

- 不改变业务分支语义
- 不改变状态流转顺序
- 不改变事件发射顺序
- 不改变用户可见错误语义
- 不改变日志 key 与关键观测点含义

### 3.2 优先提取私有 helper

如果目标只是缩短长函数、降低阅读负担，应优先提取私有 helper，而不是直接引入新的公开接口、新模块或新抽象层。

### 3.3 不为测试便利扩大可见性

测试整理应优先保持与生产代码 colocated，或继续使用嵌套测试模块。若测试依赖私有实现细节，不应仅为了迁移测试位置而将私有函数改为 `pub` 或 `pub(crate)`。

### 3.4 保持 diff 小而明确

一次结构性调整应尽量只做一类事情，避免把命名修改、业务修正、目录迁移、契约变更混在同一批 diff 中。

### 3.5 锁与异步边界不可退化

涉及并发、异步、后台任务的代码整理时，应特别确认：

- 不跨 `await` 持有锁
- 不改变 cancel / stop / worker 生命周期语义
- 不改变后台任务与回收流程的先后关系
- 不引入新的共享状态竞争

## 4. 测试整理规则

测试整理应服务于“更快定位行为覆盖”，而不是为了形式上的拆分。

建议遵循：

- 按行为域、规则域、场景域分组
- 保持测试命名直接表达行为结果
- 只提取少量必要的共享测试 helper
- 避免为了 DRY 过度抽象测试代码
- 不改变断言含义
- 涉及文件系统路径、缓存路径、下载输出路径、持久化路径时，断言不得写死平台分隔符；优先比较 `Path` / `PathBuf` 语义，或先做统一规范化后再比较
- 只在 macOS 本地通过的路径相关测试不视为充分验证；进入审批前至少要确认测试不会依赖 POSIX 风格路径表示，避免在 Windows runner 上因 `\` 与 `/` 差异产生假失败

## 5. 文档要求

如果一轮变更需要审批，应有一份单文件文档描述约束与检查项，但文档应聚焦规则和边界，而不是重复实现细节。

文档应回答：

- 这类改动允许做什么
- 不允许做什么
- 审批时重点看什么
- 验证需要覆盖什么

## 6. 最低验证要求

进入审批前，至少应满足：

- 相关测试通过
- 变更范围与提交目的匹配
- diff 可从标题和结构上快速判断边界
- 没有夹带无关逻辑修改
- 文档描述与实际改动范围一致

如果改动触达播放、下载、扫描、后台任务等编排逻辑，还应额外确认：

- 关键时序没有变化
- 资源清理路径没有变化
- 失败路径与取消路径没有变化

## 7. 审批清单

- [ ] 变更目标属于结构整理、测试整理或文档补充
- [ ] 未混入新的业务能力或契约调整
- [ ] helper / 模块命名能直接表达职责
- [ ] 测试分组比改动前更容易定位
- [ ] 未扩大生产代码可见性
- [ ] 未改变并发、取消、回收、事件、日志等关键语义
- [ ] 相关验证已完成
- [ ] 文档与实际范围一致，可直接进入审批
